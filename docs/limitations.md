# PaperPilot 系统局限性分析

> 版本：v1.6.4 | 最后更新：2026-06-08

---

## 当前版本限制（v1.6.4）

v1.6.4 已实现论文发现、前沿追踪/领域了解/深入了解三种排序模式、PDF 阅读问答、内容总结、关注领域追踪、研究趋势分析、论文关系图谱、自动文献综述、创新点分析、Reviewer 视角分析，以及科研洞察与后续选题建议。

### 1. 功能层面

| 功能 | 状态 | 说明 |
|------|------|------|
| arXiv 搜索（15s 超时） | ✅ | 超时/失败时有内置示例论文兜底 |
| 前沿追踪 / 领域了解 / 深入了解排序 | ✅ | 固定规则，可解释，不依赖 LLM；领域了解和深入了解模式使用保守启发式信号 |
| 结构化总结（LLM + Fallback） | ✅ | 支持 PDF 全文增强 |
| PDF 上传与文本解析 | ✅ | pypdf，页码标记，文本预览 |
| 论文问答（LLM + Fallback） | ✅ | 关键词检索 + LLM，引用片段，问答历史 |
| Markdown 笔记生成 | ✅ | 含 QA 历史 + 追踪摘要 |
| 关注领域管理 | ✅ | CRUD + 启用停用 |
| 论文追踪刷新 | ✅ | 手动一键刷新，去重，新论文识别 |
| 论文状态标记 | ✅ | unread/reading/read/starred/ignored |
| Reviewer 视角分析 | ✅ | LLM + Fallback，优点/风险/推荐倾向 |


### 状态隔离与一致性

- 系统通过 `paper_id` 防止 PDF、summary、QA、Reviewer 在不同论文之间误用
- Reviewer Tab 可以分析候选论文，但不会自动切换当前阅读论文（`current_paper`/`current_paper_id`）
- 手动上传 PDF 使用 `manual_pdf` 状态，不会被误识别为 hash_xxx
- 笔记生成以 `current_paper`/`current_paper_id` 为主，按 `paper_id` 过滤 summary/QA/Reviewer
- 系统无法自动验证用户上传的 PDF 内容是否真的对应当前论文



### 趋势分析限制

- 趋势分析只基于当前搜索结果和本地追踪历史；
- 不代表整个学术领域趋势；
- 关键词抽取采用轻量规则方法（unigram/bigram + 停用词过滤），不是严格 NLP 模型；
- 新兴关键词判断是启发式统计（按时间中位切分比较），不是因果分析；
- 分类分布依赖 arXiv metadata，无分类论文记为 Unknown；
- 数据量较小时（< 10 篇）趋势判断仅供参考；
- 不引入外部文献数据库；
- 主题桶基于预定义关键词规则，一篇论文可进入多个桶。


### 论文关系图谱限制

- 当前图谱主要基于论文元信息和关键词相似度；
- 如果没有 references/citations 字段，不能视为真实引用网络；
- 图谱质量依赖搜索结果覆盖度、摘要质量和分类信息；
- 样本过少时图谱仅供参考（建议 ≥ 5 篇）；
- 相似度边是启发式推断（Jaccard + 关键词），不代表论文之间存在直接引用或学术传承关系；
- 聚类是简单连通分量，不是社区发现算法；
- 中心论文是加权度度量，不等同于真实学术影响力。


### 领域了解模式的限制

- 当前领域了解模式主要基于 arXiv 元信息、标题、摘要和启发式关键词进行判断。
- 系统尚未接入真实 citation count，因此不能严格判断某篇论文是否为领域经典文献。
- 系统尚未接入完整 venue 数据库，因此权威性判断主要依赖 arXiv 的 journal_ref、comment 等元信息中的会议/期刊关键词。
- 如果 arXiv 元信息缺失，权威性信号可能不完整。
- 领域了解模式推荐的是“适合作为入门候选”的论文，不代表这些论文一定是领域最经典或最权威论文。
- 后续可通过接入 OpenAlex、Semantic Scholar、DBLP 等数据源增强 citation、venue 和经典文献识别能力。

### 深入了解模式的限制

- 深入了解模式基于论文标题、摘要、元信息、comment、journal_ref 等字段进行启发式判断，不能替代人工精读。
- v1.6.5 的相关性门控只基于用户 query、少量主题扩展词、标题、摘要和元信息做启发式匹配，不是语义检索，也不能保证覆盖所有同义表达。
- 对 `4dvar` 等少量明确缩写，系统内置了轻量扩展；其他领域缩写如果用户没有写出常见全称，仍可能漏掉相关论文。
- benchmark / evaluation / ablation / venue 等质量信号不会单独作为查询相关性证据，因此窄方向搜索更依赖用户输入的主题词质量。
- 权威会议/期刊信号主要来自 arXiv 元信息、comment、journal_ref 或已有字段，可能存在缺失或不准确。
- 系统不会联网验证论文是否真实发表在某个会议或期刊，发表信息仍需用户人工确认。
- 对 arXiv-only 论文，系统不会直接排除，只是不会给予正式发表加分。
- 系统不读取 PDF 全文来判断方法完整性、实验充分性或局限性，因此长正文中的关键信息可能被遗漏。
- 系统不验证代码仓库是否存在、是否可运行，也不判断真实复现难度。
- “值得深入阅读”是辅助排序结果，不代表系统能够真实判断论文最终学术价值。

### 2. 当前限制

| 限制 | 说明 |
|------|------|
| 手动刷新 | 需要用户手动点击刷新按钮，不支持后台定时任务 |
| arXiv 查询质量 | 追踪效果依赖关键词质量，关键词不精确可能漏掉或误召回 |
| 本地 JSON 存储 | 不适合多人协作或跨设备同步，数据量极大时性能下降 |
| arXiv ID 去重 | 新论文识别基于 arXiv ID + 标题 hash，跨数据源可能漏去重 |
| PDF 解析 | 扫描版 PDF（无文字层）无法解析；双栏/公式/表格提取可能不完整 |
| 中文检索英文 PDF | 关键词匹配可能弱于语义检索 |
| 无语义检索 | QA 和追踪均为关键词匹配，不涉及语义理解 |
| 仅 arXiv | 不覆盖 PubMed、Semantic Scholar 等其他学术数据库 |
| Reviewer 分析仅供参考 | 不是正式审稿意见，Fallback 模式较模板化；PDF 不足时置信度降低 |

### 3. 已实现但仍有改进空间的功能

- **研究趋势分析**：已实现轻量版（基于标题/摘要关键词统计和时间信息），但不是基于长期大规模学术数据库的统计分析。
- **论文关系图谱**：已实现启发式关系图谱（基于元信息/关键词相似度），但不是严格的真实引用网络。
- **自动文献综述**：已实现规则化生成（支持结合趋势分析和关系图谱），但不是完全替代人工写作的深度学术综述。

### 4. 明确不包含

以下功能在当前版本中**不包含**，也不会在近期加入：

- ❌ 后台自动定时刷新
- ❌ 用户登录 / 多用户
- ❌ 云端数据库 / 多端同步
- ❌ 向量数据库 / 语义搜索
- ❌ 复杂推荐算法
- ❌ 移动端适配

### 4. 与理想系统的差距

| 维度 | 当前 | 理想 |
|------|------|------|
| 论文来源 | arXiv only（内置示例兜底） | 多源聚合 |
| 排序 | 六维启发式规则 | 规则 + 语义匹配 + 用户反馈 |
| 总结 | 摘要 / 全文前 6000 字 | 完整全文 + 图表理解 |
| 问答 | 关键词检索 + LLM | 向量语义检索 + 精确定位引用 |
| 追踪 | 手动刷新 + 关键词去重 | 定时自动 + 语义去重 |
| 笔记 | 一次性生成 | 持续编辑 + 知识图谱关联 |
| 部署 | 本地 Streamlit | Web 服务 + 移动端 |

## 安全注意事项

- `.env` 仅用于本地运行，**严禁提交到 Git**
- 在线部署时应使用 Streamlit Cloud Secrets 配置 API Key，**不要提交 `.streamlit/secrets.toml`**
- `.gitignore` 已配置忽略 `.env`、`.streamlit/secrets.toml`、`__pycache__/`、`*.pyc`、`.DS_Store`
- 打包前请清理：`__pycache__/`、`__MACOSX/`、`*.pyc`、`.DS_Store`、`.env`、`data/notes/*.md`

### 在线 Demo 部署限制

- Streamlit Community Cloud 部署适合展示单用户可运行原型，不适合多人长期协作或高并发使用。
- 云端环境中的本地 JSON 数据可能随重启或重新部署而变化，不应作为长期数据库使用。
- arXiv 搜索依赖外部网络，如果 arXiv API 超时，系统会提示使用内置示例论文继续演示。
- 未配置 LLM API Key 时，系统会进入基础模式；LLM 摘要、问答、Reviewer 和创新点分析会降级或提示配置。
- 在线 Demo 不应包含真实 `.env`、真实 API Key 或本地个人路径。

### 在线 API 输入限制

- 在线 API 输入仅用于 Demo 场景中的临时体验。API Key 保存在当前 Streamlit 会话状态中，不会写入项目文件。
- 页面刷新、会话过期或服务重启后，用户可能需要重新输入 API Key。
- 系统不会验证所有第三方 OpenAI-compatible 服务的兼容性。如果调用失败，用户需要检查 API Key、Base URL、模型名称以及服务商接口格式。
- 页面不会明文展示完整 API Key，也不应在日志、README、`.env.example` 或 `data/*.json` 中保存真实 API Key。
